# V3 Wallet/Account Model Refactoring — Design Spec

## Goal

Clean up the V1 Wallet model for V3: remove old Prisma-based balance layer (JournalLine / WalletBalanceSnapshot / WalletBalanceEntry), adopt new wallet role taxonomy from the architecture, add `mockBalance` for simulation, split `wallets.service.ts` by responsibility, and implement System Wallet Provisioning as a manual operation during the asset PROVISIONING/ACTIVE phase.

## Background

### Two-Layer Separation

| Layer | Concern | Data Source |
|-------|---------|-------------|
| TB Accounting | Who owes whom, how much | TigerBeetle |
| Wallet Physical | Which address/account holds funds | Chain RPC / Bank API (mockBalance in simulation) |
| Reconciliation (future) | Do the two layers agree? | Compare TB vs. external (Wave 2/3) |

TB has no wallet dimension. TB accounts (CUSTODY, TRADE_CLEARING, FEE_RECEIVABLE, customer accounts) represent accounting concepts, not physical wallets. A single CUSTODY TB account may map to multiple physical wallets (hot, cold, etc.).

### V1 Problem

V1 Wallet balance is tracked by `WalletBalanceSnapshot` + `WalletBalanceEntry` + `JournalLine` — a Prisma-side double-entry system. V3 replaces this with TB for accounting. The physical wallet balance will come from external sources (chain RPC, bank API); during simulation, `mockBalance` on the Wallet model serves as a stand-in.

## 1. Prisma Schema Changes

### 1.1 Delete Models

Remove these three models and their associated DB tables:

- `JournalLine` (schema.prisma:1889-1924) — old double-entry journal lines
- `WalletBalanceSnapshot` (schema.prisma:1926-1944) — wallet balance cache
- `WalletBalanceEntry` (schema.prisma:1946-1965) — balance delta entries

### 1.2 Remove Relation Fields from Wallet

Delete from the `Wallet` model (schema.prisma:1481-1533):

```
journalLines          JournalLine[]                   @relation("JournalLineWallet")
balanceSnapshots      WalletBalanceSnapshot[]
balanceEntries        WalletBalanceEntry[]
```

### 1.3 Add mockBalance to Wallet

```prisma
mockBalance   Decimal   @default(0)
```

Simulation-phase stand-in for physical wallet balance. To be replaced by real external data source when custodian/chain integration is available.

### 1.4 Update WalletRole Enum

**Old (V1):**
```
GENERAL, DEPOSIT, MASTER, PAYOUT, LIQ, CUST_BANK, LIQ_BANK
```

**New (V3):**

| Role Code | Full Name | Owner | Asset Type | Scope |
|-----------|-----------|-------|------------|-------|
| `C_DEP` | Client Deposit | CUSTOMER | CRYPTO | Per-customer per-asset |
| `C_VIBAN` | Client vIBAN | CUSTOMER | FIAT | Per-customer per-asset |
| `C_MAIN` | Client Omnibus | PLATFORM | CRYPTO | Per-asset system wallet |
| `C_OUT` | Client Outbound | PLATFORM | CRYPTO | Per-asset system wallet |
| `C_CMA` | Client Money Account | PLATFORM | FIAT | Per-asset system wallet |
| `F_LIQ` | Company Liquidity | PLATFORM | BOTH | Per-asset system wallet |
| `F_OPS` | Company Operations | PLATFORM | BOTH | Per-asset system wallet |

Update the TypeScript enum in `dto/wallet.dto.ts` and the Prisma schema default value.

### 1.5 Models NOT Touched

- `Wallet` model itself (all non-balance fields and relations remain: DepositTransaction, Payin, InternalTransaction, InboundTransferSignal, etc.)
- `Journal`, `JournalLineTemplate`, `Coa` — old accounting models outside this refactoring scope

## 3. System Wallet Provisioning

### 3.1 Trigger

Manual operation by admin. New API endpoint:

```
POST /admin/assets/:assetNo/provision-wallets
```

Requires `@RequirePermissions(buildPermissionCode('POST', '/admin/assets/:assetNo/provision-wallets'))`.

### 3.2 State Constraints

| Operation | Required Asset Status |
|-----------|---------------------|
| Create TB accounts | PENDING_APPROVAL → PROVISIONING (automatic on approval) |
| Create system wallets | PROVISIONING or ACTIVE (manual) |
| Activate asset | PROVISIONING → ACTIVE (manual) |
| Customer creates deposit wallet | ACTIVE only (separate workflow) |

System wallets can be added after ACTIVE — not locked to provisioning phase.

### 3.3 Creation Logic

```
provisionSystemWallets(asset):
  if asset.type === 'CRYPTO':
    create wallets with roles: C_MAIN, C_OUT, F_LIQ, F_OPS
  if asset.type === 'FIAT':
    create wallets with roles: C_CMA, F_LIQ, F_OPS
```

Each wallet:
- `walletNo`: `generateReferenceNo('WA')`
- `ownerType`: PLATFORM
- `role`: from the list above
- `assetId`: the asset being provisioned
- `mockBalance`: 0
- `status`: ACTIVE
- `type`: CRYPTO_ADDRESS (for crypto) / FIAT_BANK (for fiat)
- `direction`: BIDIRECTIONAL

Idempotency: if a wallet with the same (role, assetId, ownerType=PLATFORM) already exists, skip it. This allows partial re-runs and adding wallets after ACTIVE.

### 3.4 Audit Logging

Record via `AuditLogsService.recordByActor()`:
- action: `ASSET_LISTING.SYSTEM_WALLETS_PROVISIONED`
- entityType: ASSET
- metadata: list of created wallet roles and walletNos

## 4. Old Balance Layer Cleanup

### 4.1 Files in Refactoring Scope

These files are rewritten as part of this refactoring — old balance logic is completely replaced:

- `src/modules/asset-treasury/wallets/wallets.service.ts`
- `src/modules/asset-treasury/wallets/dto/wallet.dto.ts`

### 4.2 Files Outside Scope — Runtime Deprecation

These files reference `JournalLine`, `WalletBalanceSnapshot`, or `WalletBalanceEntry`. The balance-related code segments are replaced with runtime exceptions:

```typescript
throw new Error('DEPRECATED: migrate to TB — [original function description]');
```

Method signatures and imports are preserved so the code compiles. The runtime error surfaces when the code path is actually hit.

## 6. Migration Notes

### 6.1 Prisma Migration

A single migration that:
1. Adds `mockBalance` column to `wallets` table
2. Drops `wallet_balance_entries` table
3. Drops `wallet_balance_snapshots` table
4. Drops `journal_lines` table (if no other non-wallet dependents block this — check Journal model relation)

**Cascading relation cleanup:** Dropping `JournalLine` requires removing reverse-relation fields from other models:

| Model | Field to Remove |
|-------|----------------|
| `Journal` | `lines JournalLine[]` |
| `JournalLineTemplate` | relation to JournalLine (if exists) |
| `Wallet` | `journalLines`, `balanceSnapshots`, `balanceEntries` |
| `Asset` | reverse relation to JournalLine, WalletBalanceSnapshot, WalletBalanceEntry |
| `Coa` | reverse relation to JournalLine |

The `Journal`, `JournalLineTemplate`, `Coa` models themselves are kept (old accounting infrastructure — separate cleanup scope) but lose their JournalLine relations.

### 6.2 WalletRole Migration

Existing wallet rows with old role values (MASTER, PAYOUT, etc.) need a data migration:

| Old Role | New Role |
|----------|----------|
| DEPOSIT | C_DEP |
| MASTER | C_MAIN |
| PAYOUT | C_OUT |
| LIQ | F_LIQ |
| CUST_BANK | C_CMA |
| LIQ_BANK | F_LIQ (with fiat asset context) |
| GENERAL | Evaluate per-wallet — likely C_DEP or remove |

This mapping runs as part of the Prisma migration (SQL UPDATE statements).
