# Admin Role Binding Change — 3-Layer Workflow Design

## Problem

The roadmap defines "Admin Role Binding Change" as a dedicated approval workflow with SoD conflict checking. The current implementation routes role changes through the **Change Ticket** mechanism (`RBAC_CATALOG_CHANGE` ticket type → `GovernedExecutionListener`), which:

- Shares a generic approval type (`CHANGE_TICKET_APPROVAL`) with unrelated config changes
- Has no dedicated audit action namespace (uses generic change ticket audit events)
- Cannot enforce role-binding-specific pre-flight validation (e.g., SoD hard mutex, self-change prohibition)
- Doesn't produce a queryable request entity for admin UI display

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture | Independent 3-layer workflow (like Admin Invite) | Pattern consistency, dedicated audit namespace |
| Entity | New `AdminRoleChangeRequest` Prisma model | Need to hold before/after role snapshots until approved |
| Old path | Replace — delete Change Ticket role binding code | One canonical path prevents audit chain confusion |
| Session revocation | Out of scope | Future Session Force-Revocation workflow handles this |
| SoD validation | Pre-flight + execution double check | Prevents race condition during approval wait period |
| Self-change | Prohibited | `targetUserId === actor.userId` rejected at request time |
| Soft warnings | Not used | Only hard mutex enforcement; `SOFT_WARNING_ROLE_GROUPS` stays empty |

## Design

### Data Model — `AdminRoleChangeRequest`

New Prisma model in `schema.prisma`:

```prisma
model AdminRoleChangeRequest {
  id                String    @id @default(uuid())
  requestNo         String    @unique @default("TEMP")
  targetUserId      String
  currentRoleCodes  String    // JSON array snapshot at request time
  proposedRoleCodes String    // JSON array of target roles
  changeReason      String
  status            String    @default("PENDING_APPROVAL")
  approvalCaseId    String?
  approvalCaseNo    String?
  requestedByUserId String
  executedAt        DateTime?
  failureReason     String?
  createdAt         DateTime  @default(now())
  updatedAt         DateTime  @updatedAt
  deletedAt         DateTime?

  targetUser        User      @relation("roleChangeTarget", fields: [targetUserId], references: [id])

  @@index([targetUserId])
  @@index([approvalCaseId])
  @@index([status])
  @@index([requestedByUserId])
  @@map("admin_role_change_requests")
}
```

Status enum values: `PENDING_APPROVAL`, `APPROVED`, `REJECTED`, `CANCELLED`, `EXPIRED`, `FAILED`.

Status after successful execution: stays `APPROVED` with `executedAt` set. The `FAILED` status is only used when execution throws after approval.

`requestNo` format: `RCR-{YYMMDD}{seq}` (e.g., `RCR-2605050001`), generated by the same sequence utility used for other business numbers.

**User model change:** Add reverse relation field `roleChangeRequests AdminRoleChangeRequest[] @relation("roleChangeTarget")` to the `User` model.

### SoD Hard Mutex Rules

Populate `HARD_MUTEX_ROLE_PAIRS` in `rbac.catalog.ts`:

```typescript
export const HARD_MUTEX_ROLE_PAIRS: Array<[string, string]> = [
  ['CISO', 'MLRO'],
  ['MLRO', 'OPS_OFFICER'],
  ['CISO', 'OPS_OFFICER'],

```

`SOFT_WARNING_ROLE_GROUPS` remains empty.

Enforcement: `AccessControlService.validateHardMutex()` already reads this array and throws `BadRequestException` on conflict. No new validation logic needed — just populate the data.

### Architecture

```
Controller: POST /admin/iam/role-change-requests
    │
    ▼
Layer 3: AdminRoleBindingChangeWorkflowService
    │ calls domain methods         subscribes to decided event
    ▼                                       ▲
Layer 1: AccessControlService      Layer 2: AdminRoleBindingChangeApprovalService

                                   writes approval audit + emits decided event
```

### Layer 3 — Workflow Orchestrator

File: `src/modules/identity/users/admin-role-binding-change-workflow.service.ts`

**Dependencies:** `AccessControlService`, `ApprovalsService`, `AuditLogsService`, `PrismaService`

**Public methods:**

| Method | Trigger | Audit Log |
|--------|---------|-----------|
| `createRoleChangeRequest(dto, actor)` | Controller POST | `CHANGE_REQUESTED` |
| `findRoleChangeRequests(query)` | Controller GET list | (none) |
| `findRoleChangeRequest(id)` | Controller GET detail | (none) |

**Event handler:**

```typescript
@OnEvent('workflow.admin-role-binding-change.decided', { async: true })
async handleApprovalDecided(event: ApprovalDecidedEvent)
```

| Internal method | Action | Audit Log |
|----------------|--------|-----------|
| `executeRoleChange(event)` | Validate SoD again, call `replaceUserRoles()`, update request | `CHANGE_APPLIED` or `CHANGE_APPLY_FAILED` |
| `executeTermination(event)` | Update request status to REJECTED/CANCELLED/EXPIRED | (none — Layer 2 governance log suffices) |

**`createRoleChangeRequest` flow:**
1. Validate `targetUserId !== actor.userId` — reject self-change
2. Load target user, verify exists and not deleted
3. `accessControlService.getUserRoleCodes(targetUserId)` — snapshot current roles
4. `accessControlService.validateHardMutex(proposedRoleCodes)` — pre-flight SoD check (throws on conflict)
5. Create `AdminRoleChangeRequest` record (status = `PENDING_APPROVAL`)
6. `approvalsService.createAndSubmit(...)` — create + submit approval case
7. Link `approvalCaseId` / `approvalCaseNo` to request
8. `auditLogsService.recordByActor(CHANGE_REQUESTED, ...)` — audit trail
9. Return request detail

**`executeRoleChange` flow (APPROVED):**
1. Load `AdminRoleChangeRequest` by approval binding
2. Parse `proposedRoleCodes` from stored JSON
3. Call `accessControlService.replaceUserRoles(targetUserId, proposedRoleCodes, actor)` — this internally re-validates hard mutex
4. Update request: `executedAt = now()`
5. `auditLogsService.recordByActor(CHANGE_APPLIED, ...)`
6. `approvalsService.markExecutionResult(approvalId, true, ...)`
7. On error: update request `status = FAILED`, `failureReason`, write `CHANGE_APPLY_FAILED`, `markExecutionResult(false)`

**`executeTermination` flow (DECLINED/CANCELLED/EXPIRED):**
1. Load request by approval binding
2. Update status to `REJECTED` / `CANCELLED` / `EXPIRED`

### Non-Goals

- No session revocation on role change (future Session Force-Revocation workflow)
- No SoD Rules admin UI tab (supporting feature, separate delivery)
- No changes to `ApprovalHandlerBase` or `ApprovalsService` internals
- No soft warning rules (only hard mutex)
- No migration of historical Change Ticket role binding records
